## 背景

TabTrail 当前的三个界面入口职责已经基本明确：

- popup 是轻量入口，负责打开侧边栏、展示摘要、展示最近活跃和最近关闭。
- side panel 是浏览过程中常驻的整理工作台，负责当前窗口/全部窗口范围内的搜索、选择、排序、归类和关闭。
- newtab 是空间更大的默认仪表盘，负责分类优先的整理视图、最近记录、批量操作和关闭分类。

本次 UX 审查暴露的问题不是单个页面的视觉问题，而是跨入口的交互契约不一致：范围状态不够清楚，动态反馈不稳定，关闭操作没有可恢复机制，拖拽路径没有完整键盘替代，列表重渲染后焦点可能丢失。这些问题会直接影响重度标签用户、键盘用户、屏幕阅读器用户和低视力用户。

项目当前使用原生 HTML/CSS/JS，不引入前端框架。设计应沿用现有模块边界和测试方式，优先用小型共享 helper 统一交互行为，而不是大规模重写页面。

## 目标 / 非目标

**目标：**

- 让用户始终能看懂当前整理范围是“当前窗口”还是“全部窗口”。
- 让关闭、归类、排序、保存失败、空结果等状态有稳定的可见反馈和读屏公告。
- 让标签整理核心任务不依赖鼠标拖拽，键盘用户能完成选择、排序、移动分类和关闭。
- 让列表重渲染后焦点可预测，支持连续键盘操作。
- 让破坏性操作可恢复，降低误关标签的心理成本。
- 让 popup、side panel、newtab 在相同概念上使用一致文案、控件语义和反馈模型。
- 用测试锁定新的交互要求，避免后续 UI 调整破坏无障碍和恢复路径。

**非目标：**

- 不重新设计产品定位，不移除现有 popup、side panel 或 newtab 任一入口。
- 不引入 React/Vue/Svelte 等框架。
- 不引入后端服务、云同步、远程埋点或外部可用性分析 SDK。
- 不改变最近关闭只读系统分类的原则。
- 不修改浏览器默认搜索引擎、主页或地址栏搜索行为。
- 不承诺直接隐藏浏览器原生顶部标签栏。

## 关键决策

### 决策：范围切换使用分段按钮，而不继续伪装成 tabs

当前 side panel 和 newtab 使用 `role="tablist"` 表达“当前窗口 / 全部窗口”。但这两个按钮并不切换 tab panel，也没有完整 tab 键盘模型。继续补齐 tabs 语义会把简单范围切换做复杂。

采用分段按钮模型：每个按钮使用 `aria-pressed="true/false"` 表达选中状态，控件容器使用明确标签“标签范围”。摘要区必须显示当前范围文案，例如“当前窗口：12 个标签”或“全部窗口：38 个标签”。

替代方案是完整实现 tabs：`role="tab"`、`aria-controls`、方向键切换、tabpanel。该方案适合内容面板切换，但这里是查询范围切换，语义成本更高。

### 决策：每个入口使用专用状态公告节点

动态反馈不应散落在列表标题、按钮文案或整个列表 live region 中。每个入口都应有一个稳定状态节点：

- 普通状态：`role="status" aria-live="polite" aria-atomic="true"`。
- 错误或无法继续：使用 `role="alert"` 或同一节点切换为错误语义。
- 列表区域不作为整体 live region，避免搜索或重渲染时读出大量标签内容。

状态节点只公告用户关心的结果，例如“已归类 3 个标签到 工作”“排序已应用到当前窗口”“没有匹配标签，已显示清空搜索按钮”。

替代方案是继续复用现有说明文案节点。该方案改动少，但读屏公告不稳定，也容易把说明文案和操作结果混在一起。

### 决策：关闭操作采用“确认 + 撤销/恢复反馈”双层保护

关闭标签是高风险操作，但每次关闭 1 个标签都弹确认会降低效率。设计采用双层策略：

- 关闭 3 个及以上标签、关闭分类、全部窗口范围下的批量关闭必须确认。
- 所有关闭成功后都必须展示恢复反馈：优先提供短时撤销入口；若浏览器 API 或当前模型无法原位撤销，则展示“可从最近关闭重新打开”的明确恢复路径。
- 状态反馈必须包含数量和范围，例如“已关闭当前窗口中的 2 个标签，可从最近关闭恢复”。

替代方案是所有关闭都确认。该方案更保守，但会让高频整理流程过重。

### 决策：拖拽只作为增强路径，键盘路径必须完整

newtab 的分类拖拽和 side panel 的排序拖拽适合鼠标用户，但不能作为唯一方式。每个可整理标签至少需要以下非拖拽路径之一：

- 行内上移/下移按钮。
- “移动到分类”按钮或菜单。
- 选择多个标签后输入分类名并点击“归类已选”。

拖拽成功、不可放置原因、键盘移动结果都统一写入状态公告节点。

替代方案是仅补充拖拽说明文案。该方案不能解决键盘和触控可用性。

### 决策：重渲染前后记录并恢复焦点

现有 side panel 和 newtab 在选择、排序、搜索、归类后会重建 DOM。实现时应为行内控件添加稳定标识，例如 `data-tab-id` 和 `data-control`，在触发操作前记录当前焦点，渲染完成后优先恢复到同一 tab 的同类控件；如果目标已消失，则恢复到合理后继位置或状态区域。

替代方案是大规模改为局部 DOM 更新。该方案长期更理想，但本变更应先保证行为正确，再按风险逐步优化局部更新。

### 决策：排序引入显式 dirty 状态

“应用排序”必须表达当前预览是否已经与浏览器标签栏一致。设计引入轻量 dirty 状态：

- 初始或已应用：按钮禁用或显示“排序已同步”。
- 用户上移/下移/拖拽后：按钮启用，状态提示“有未应用排序”。
- 应用成功：公告“排序已应用”。
- 应用失败：公告失败原因，并保留预览状态。

替代方案是保持按钮一直可点。该方案实现简单，但用户无法判断是否需要应用。

## 风险 / 取舍

- 风险：新增状态公告和撤销反馈让 UI 更复杂。缓解：使用统一组件样式和短文案，只在操作后显示。
- 风险：短时撤销需要保存关闭前的 URL/窗口上下文，浏览器无法完全恢复原标签状态。缓解：第一版至少提供 URL 级恢复，并在文案中避免承诺完整状态恢复。
- 风险：焦点恢复逻辑容易产生边界 bug。缓解：用测试覆盖选择、排序、搜索无结果、关闭后列表变化等关键路径。
- 风险：键盘移动到分类菜单可能增加每行控件密度。缓解：在 side panel 使用紧凑图标按钮配合 `aria-label`，在 newtab 使用行内菜单或批量工具区。
- 风险：全部窗口范围下的关闭操作仍可能让用户紧张。缓解：确认文案必须包含“全部窗口”和数量，并保留恢复说明。

## 迁移计划

1. 先增加共享状态公告、焦点恢复和范围控件约定，不改变业务数据结构。
2. 重构 popup 状态反馈，确保打开侧边栏、偏好保存和初始化失败有稳定公告。
3. 重构 side panel 范围切换、键盘排序、批量归类、关闭反馈和焦点恢复。
4. 重构 newtab 分类切换、移动到分类、关闭分类、应用排序和空状态语义。
5. 更新测试，覆盖新交互契约。
6. 如实现中引入撤销队列，数据只保存在当前页面运行时或本地最近关闭记录中，不做远程迁移。

回滚策略：本变更不迁移持久数据。若某入口出现严重问题，可回退对应 HTML/CSS/JS 交互改动，保留现有存储格式。

## 待确认问题

- 关闭撤销是否需要恢复到原窗口和原索引，还是第一版只恢复 URL 即可？
- “移动到分类”在 side panel 中采用行内菜单、弹出菜单，还是复用批量归类输入作为主要键盘路径？
- 搜索 debounce 默认使用 100ms 还是 150ms，需要通过大量标签场景手测确认。
- newtab 的“最近关闭”是否保留在分类网格中，还是只保留在右侧最近记录区以减少重复？
